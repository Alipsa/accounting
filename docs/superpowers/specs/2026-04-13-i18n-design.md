# i18n Design: English and Swedish Language Support

## Overview

Add internationalization support to Alipsa Accounting with English and Swedish as the two supported languages. The implementation uses standard Java `ResourceBundle` with a thin observer layer for runtime locale switching without application restart.

## Decisions

- **Base language:** English in `.properties` files; Swedish as overlay.
- **Runtime switching:** Supported without restart, via `PropertyChangeListener` pattern.
- **Locale persistence:** Default to `Locale.getDefault()` on first launch, persist choice in `CompanySettings.localeTag`.
- **Enum labels:** Remove hardcoded Swedish label fields; derive all display names from resource bundles using enum name as key.
- **Scope:** Only UI-visible text is translated. Log messages, exception messages, and stack traces remain in English for developer debugging.
- **Key convention:** Hierarchical dot-notation: `className.context.name` (e.g., `mainFrame.menu.file`, `vatCode.OUTPUT_25`).
- **Language selector:** Two flag buttons (`icons/sweden.png`, `icons/UK.png`) in `CompanySettingsDialog`.

## Architecture

### Resource Bundle Files

```
app/src/main/resources/i18n/
  messages.properties        # English (base/fallback)
  messages_sv.properties     # Swedish
```

Standard Java `.properties` format with UTF-8 encoding.

### `I18n` Utility Class

Location: `se.alipsa.accounting.support.I18n`

```
I18n (singleton)
  - currentLocale: Locale
  - bundle: ResourceBundle
  - propertyChangeSupport: PropertyChangeSupport

  + getString(key): String
  + getString(key, ...args): String       # MessageFormat for parameterized strings
  + setLocale(locale): void               # Reloads bundle, fires "locale" change event
  + getLocale(): Locale
  + addLocaleChangeListener(listener): void
  + removeLocaleChangeListener(listener): void
```

- Wraps `ResourceBundle.getBundle("i18n.messages", locale)` for string loading.
- `setLocale()` clears the ResourceBundle cache, reloads the bundle, and fires a `PropertyChangeEvent` with property name `"locale"`.
- `getString(key, ...args)` uses `MessageFormat.format()` for parameterized messages like `"Fiscal year {0} created."`.

### Initialization Sequence

1. `AlipsaAccounting.main()` creates `I18n` with `Locale.getDefault()` before any UI.
2. `DatabaseService.initialize()` runs.
3. `CompanySettings.localeTag` is loaded. If set, `I18n.instance.setLocale()` is called with the stored locale. No UI listeners exist yet, so no repaint occurs.
4. UI components are created and register as locale change listeners.

## Enum Translation

Remove Swedish label fields from `VatCode`, `VatPeriodicity`, and `ReportType`. Replace with a `getDisplayName()` method that looks up translations via `I18n`.

### Before

```groovy
enum VatPeriodicity {
  MONTHLY('Månadsvis'),
  ANNUAL('Årsvis')
  final String label
  VatPeriodicity(String label) { this.label = label }
}
```

### After

```groovy
enum VatPeriodicity {
  MONTHLY,
  ANNUAL

  String getDisplayName() {
    I18n.instance.getString("vatPeriodicity.${name()}")
  }
}
```

### Affected Enums

| Enum | Values | Fields Kept | Fields Removed |
|------|--------|-------------|----------------|
| `VatCode` | 13 | `rate`, `deductionRate` | `label` |
| `VatPeriodicity` | 2 | none | `label` |
| `ReportType` | 7 | `templateName`, `pdfCapable` | `displayName` |

Callers accessing `.label` or `.displayName` continue using `getDisplayName()` with no call-site changes beyond recompilation.

## UI Runtime Switching

### Pattern

Each UI panel and dialog implements `PropertyChangeListener`, subscribes to `I18n` locale changes, and re-applies all strings in an `applyLocale()` method:

```groovy
class FiscalYearPanel extends JPanel implements PropertyChangeListener {

  FiscalYearPanel() {
    applyLocale()
    I18n.instance.addLocaleChangeListener(this)
  }

  void propertyChange(PropertyChangeEvent event) {
    applyLocale()
  }

  private void applyLocale() {
    nameLabel.text = I18n.instance.getString('fiscalYearPanel.label.name')
    createButton.text = I18n.instance.getString('fiscalYearPanel.button.create')
    // etc.
  }
}
```

### Component Types

| Component | How it updates |
|-----------|---------------|
| `JLabel`, `JButton` | `.setText()` |
| `JMenu`, `JMenuItem` | `.setText()` |
| Table column headers | `TableColumn.setHeaderValue()` + `tableHeader.repaint()` |
| Dialog titles | `JDialog.setTitle()` |
| `JOptionPane` messages | Read from `I18n` at display time; no listener needed |
| Service validation messages | Call `I18n.getString()` at message creation time |

### Language Selector

Two `JButton`s in `CompanySettingsDialog` using the existing flag icons (`icons/UK.png`, `icons/sweden.png`). Clicking a button:

1. Calls `I18n.instance.setLocale(newLocale)`.
2. Persists the locale to `CompanySettings.localeTag` via `CompanySettingsService`.
3. All subscribed UI components receive the change event and re-apply labels.

The active language button gets a visual indicator (e.g., raised/lowered border).

## Scope: Translatable Strings

| Category | Count | Notes |
|----------|-------|-------|
| UI labels, buttons, menus | ~174 | Panels, dialogs, MainFrame |
| Enum display labels | 22 | VatCode (13), ReportType (7), VatPeriodicity (2) |
| User-facing service messages | ~72 | Validation errors, confirmations |
| JOptionPane messages | 5 | Startup errors, confirmations |
| **Total** | **~273** | |

Not translated: log messages, exception messages for developers, internal error strings.

## Testing Strategy

- **Unit tests for `I18n`:** `getString()` returns correct values for both locales, parameterized messages work, locale switching fires events, missing keys fall back to the English base bundle.
- **Unit tests for enum display names:** Each enum value resolves a display name in both locales.
- **Resource bundle completeness test:** Loads both `.properties` files and asserts they have identical key sets. Catches missing translations early.
- **No UI-level i18n tests:** Runtime switching is a Swing repaint concern best verified visually.
