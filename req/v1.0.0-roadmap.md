# Projektplan: Bokföringsprogram i Groovy (v1.0.0)

Den här planen avser ett komplett bokföringsprogram för ett företag per databasinstallation, med fokus på svensk löpande bokföring, moms, periodstängning, bokslut och SIE-utbyte.

## Målbild och avgränsning

### I scope för v1.0.0

- Kontoplan enligt BAS med import från Excel
- Löpande bokföring med verifikationer och bilagereferenser
- Flera räkenskapsår med ingående och utgående balanser
- Momsperioder, momsrapport och periodlåsning
- Grundbok, huvudbok, verifikationslista, provbalans, resultatrapport och balansrapport
- SIE4-import och -export
- Bokslutsflöde och överföring till nästa räkenskapsår
- Revisionsspår, behandlingshistorik, backup och arkivering
- PDF-rapporter via Journo
- Distributioner för Windows, Linux och macOS

### Utanför scope för v1.0.0

- Lön
- Fakturering
- Bankintegration
- E-faktura
- Koncernredovisning
- Full fleranvändarhantering med behörighetsroller
- Dimensioner som kostnadsställe och projektuppföljning

V1 ska alltså vara ett komplett bokföringssystem för kärnbokföring, men inte ett helt affärssystem.

- **Groovy** 5.0.5
- **JVM** 21
- **GUI** Groovy Swing med SwingBuilder
- **Databas** H2 (inbyggd fil-databas)
- **Persistens** Groovy SQL
- **Rapporter** Journo Runtime 0.7.3 med Freemarker-mallar och PDF-rendering
- **SIE** `SieParser` 2.0
- **Excel-import** Apache POI
- **Test** JUnit 6.0.3 kompletterat med `se.alipsa.groovy:groovier-junit:0.3.1` för enhets-, integrations-, acceptans- och golden-master-tester
- **Distribution** `jpackage` med plattformsspecifika launchers och installerpaket
- **Styrande regelverk** Bokföringslagen (BFL 1999:1078), BFNAR och god redovisningssed

### `Voucher.groovy`

```groovy
@Canonical
class Voucher {
    String voucherSeries       // A, B, C ...
    int fiscalYear
    int runningNumber
    String voucherNumber       // t.ex. A-2026-0001
    LocalDate voucherDate
    LocalDate bookingDate
    String description
    String attachmentRef
    String status              // DRAFT, BOOKED, CANCELLED, CORRECTION
    String correctionOfVoucher
    List<VoucherLine> lines = []

    boolean isBalanced() {
        if (lines.isEmpty()) {
            return false
        }
        BigDecimal debit = lines.inject(BigDecimal.ZERO) { sum, line -> sum + (line.debit ?: BigDecimal.ZERO) }
        BigDecimal credit = lines.inject(BigDecimal.ZERO) { sum, line -> sum + (line.credit ?: BigDecimal.ZERO) }
        debit == credit
    }
}
```

## Journo i stället för JasperReports

JasperReports tas helt bort ur planen. Rapportdelen byggs i stället på Journo med följande design:

### Rapportarkitektur

- `ReportDataService` hämtar och aggregerar data från databasen till rena DTO:er eller `Map<String, Object>`
- `JournoReportService` kapslar `JournoEngine`
- `ReportExportService` ansvarar för CSV-export av tabellrapporter
- rapportmallar lagras som Freemarker-filer under `app/src/main/resources/reports/`
- gemensam layout och utskrifts-CSS delas via `layout/base.ftl` och `layout/styles.css`
- rapporter renderas först till HTML och därefter till PDF via `JournoEngine.renderPdf(...)`

### Krav på rapportfunktionen

- samma datakälla ska kunna användas för både UI-tabeller och PDF-export
- varje rapport ska kunna köras för datumintervall, period eller helt räkenskapsår
- PDF-filen ska kunna sparas till disk och registreras i ett rapportarkiv
- tabellrapporter ska kunna exporteras till CSV med samma urval och sortering som i UI:t
- rapportarkivet ska spara körparametrar, skapad tidpunkt och checksumma
- rapporter för låsta perioder ska kunna återskapas identiskt från samma dataurval

### Rapporter som måste finnas i v1

- `VoucherList` för kronologisk verifikationslista
- `GeneralLedger` för konto-för-konto med ingående saldo, transaktioner och utgående saldo
- `TrialBalance` för saldobalans / provbalans
- `IncomeStatement`
- `BalanceSheet`
- `VatReport`
- `TransactionReport`

Journo passar bättre här än JasperReports eftersom rapporterna i detta projekt främst är dokumentgenerering från egen domändata, inte behov av en separat designer eller tung rapportserver.

## Funktionella krav

### 1. Företagsinställningar

- företagsnamn, organisationsnummer och basvaluta
- sökväg för bilagearkiv och backup-katalog
- standardserie för verifikationer
- uppsättning av första räkenskapsåret

### 2. Räkenskapsår och perioder

- flera räkenskapsår ska kunna finnas i samma databas
- stöd för brutna räkenskapsår, till exempel maj-april
- varje räkenskapsår ska delas i perioder, normalt månadsvis
- perioder ska kunna låsas separat
- stängt räkenskapsår får inte längre bokföras på
- nästa räkenskapsår ska kunna skapas från föregående års utgående balans

### 3. Kontoplan

- import av BAS-kontoplan från Excel
- manuell redigering av kontonamn, aktiv/inaktiv och momskod
- stöd för spärr mot bokning på inaktiva konton
- stöd för kontosökning, filtrering och kontoöversikt
- möjlighet att sätta ingående balans per balanskonto och räkenskapsår
- BAS-import ska härleda `accountClass` från första siffran i kontonumret; `1xxx` = `ASSET`, `2xxx` = `LIABILITY` eller `EQUITY` beroende på undergrupp, `3xxx-8xxx` = `INCOME` eller `EXPENSE` enligt BAS-huvudgrupp
- importregler som inte kan härledas säkert ska flaggas för manuell granskning

### 4. Verifikationer

- konfigurerbara verifikationsserier, minst serie `A`
- obruten löpnumrering per serie och räkenskapsår
- statusflöde: `DRAFT -> BOOKED -> CANCELLED` eller `CORRECTION`
- bokförd verifikation får inte redigeras i stället för att korrigeras
- korrigering ska ske genom korrigeringsverifikation med tydlig koppling till original
- obligatoriska fält: serie, löpnummer, verifikationsdatum, bokföringsdatum, beskrivning
- stöd för bilagsreferens och länk till lagrad bilagefil
- radvalidering: endast debet eller kredit per rad
- verifikationsvalidering: summa debet måste vara lika med summa kredit
- verifikationsdatum i låst period ska nekas

### 5. Bilagor

- metadata om bilagor ska lagras i databasen
- filerna ska sparas i ett separat arkiv på disk
- checksumma ska beräknas för att upptäcka manipulation
- bokförd verifikation ska inte kunna förlora sin bilagereferens utan revisionsspår

### 6. Moms

- momskoder per konto och möjlighet till manuell justering på verifikationsnivå
- momsperioder med status `OPEN`, `REPORTED`, `LOCKED`
- momsrapport per period
- bokning av momsöverföring mot redovisningskonto
- stöd för omvänd skattskyldighet och EU-förvärv/EU-försäljning i momskodningen
- spärr mot ändringar i momsrapporterad period utan korrigeringsflöde

### 7. Rapporter

- verifikationslista
- huvudbok
- provbalans
- resultatrapport
- balansrapport
- momsrapport
- transaktionsrapport
- utskrift till PDF via Journo
- export till CSV för tabellbaserade rapporter
- CSV-export ska initialt omfatta `VoucherList`, `GeneralLedger`, `TrialBalance` och `TransactionReport`
- rapportvyer ska där det är relevant stödja drill-down från rapportrad till underliggande verifikation

### 8. SIE-import och -export

- import av minst `#KONTO`, `#IB`, `#UB`, `#VER`, `#TRANS`
- export av aktivt räkenskapsår till SIE4
- SIE4-export ska alltid skriva obligatoriska metadata- och header-poster: `#FLAGGA`, `#FORMAT`, `#SIETYP`, `#PROGRAM`, `#GEN`, `#FNAMN`, `#ORGNR`, `#RAR`
- mappning mellan SIE och intern domänmodell
- importlogg med varningar och fel
- skydd mot dubblettimport av samma fil

### 9. Bokslut

- stängning av resultatkonton mot eget kapital, normalt konto `2099`
- generering av bokslutsverifikationer
- överföring av utgående balanser till nästa års ingående balanser
- låsning av avslutat år
- kontroll att inga öppna perioder finns kvar före årsstängning

### 10. Drift, backup och återställning

- manuell backup av databas, bilagearkiv och rapportarkiv
- backup-format ska vara ett versionsmärkt ZIP-arkiv med H2 `SCRIPT`-dump, manifest, checksummor och referenser till bilagefiler
- återställning från backup
- schema-migrationer mellan versioner
- migrationer ska vara framåtriktade och versionsstyrda via `schema_version`; rollback i produktion ersätts av restore från backup
- enkel systemdiagnostik: databasplats, storlek, senaste backup, senaste SIE-export
- systemet ska förhindra eller hårt varna vid rensning av bokföringsdata, bilagor eller rapportarkiv som omfattas av sju års bevarandekrav

### 11. Validering och felhantering i UI

- valideringsfel ska visas nära berört fält samt i en sammanfattningsruta före sparning
- affärsregelfel ska förklaras med orsak och föreslagen åtgärd, till exempel låst period eller obalanserad verifikation
- import- och exportfel ska kunna sparas i logg och visas i egen resultatdialog
- oväntade fel ska loggas och presenteras utan att dölja teknisk grundorsak
- UI:t ska använda ett gemensamt valideringsmönster så att fel presenteras konsekvent i alla paneler och dialoger

### 12. Systemdokumentation och bevarande

- systemet ska ha inbyggd systemdokumentation som kan visas i UI och exporteras för revision
- systemdokumentationen ska täcka nummerserier, tabeller, bokföringsflöden, import/export, backup/restore och integritetskontroller
- systemet ska kunna visa aktuell schema-version och aktiva migreringar
- systemet ska kunna visa och exportera revisionsrelevant metadata för backup, rapportarkiv och importjobb

### 13. Användarmanual och projektinformation

- systemet ska ha en separat användarmanual riktad till slutanvändare
- användarmanualen ska vara tillgänglig från appens hjälpmeny eller motsvarande UI
- användarmanualen ska minst täcka installation, första uppstart, företagsinställningar, kontoplan, verifikationer, moms, rapporter, SIE, backup/restore och bokslut
- användarmanualen ska kunna uppdateras versionerat tillsammans med applikationen
- `README.md` ska hållas uppdaterad med projektöversikt, bygginstruktioner, utvecklingsflöde och releaseinformation

### 14. Distribution och installation

- systemet ska kunna paketeras för Windows, Linux och macOS
- varje distribution ska innehålla startbar launcher för rätt plattform
- Windows-distributionen ska skapa genväg för applikationen
- Linux-distributionen ska innehålla launcher och `.desktop`-fil
- macOS-distributionen ska skapa `AlipsaAccounting.app`
- distributionerna ska kunna bära med standardikoner, versionsinformation och applikationsnamn
- distributionerna ska kunna använda samma konfigurationsstrategi för datakatalog och loggar, med plattformsspecifika standardvägar
- build-flödet ska kunna producera plattformspaket reproducerbart från versionsmärkt källa

## Säkerhet och manipulationsskydd

Eftersom systemet använder en lokal H2-fil kan direkt manipulation aldrig elimineras helt om en användare har filsystemsåtkomst, rättigheter och eventuell dekrypteringsnyckel. Målet för v1 är därför att göra extern manipulation operativt svår, tydligt otillåten och tekniskt upptäckbar.

### Säkerhetsprinciper

- H2 ska köras endast i embedded mode
- H2 TCP-server och H2 Console ska vara avstängda
- anslutningssträngen ska inte använda `AUTO_SERVER=TRUE`
- databasfiler ska ligga i en app-privat katalog med hårda filrättigheter
- databasen ska krypteras och krypteringshemlighet ska inte lagras i samma katalog som databasfilen
- bokförda verifikationer ska vara append-only ur affärslogikens perspektiv
- kritiska operationer ska alltid loggas i `audit_log`

### Tekniska krav

- rekommenderad H2-konfiguration ska motsvara embedded drift, till exempel `jdbc:h2:file:...;AUTO_SERVER=FALSE;DB_CLOSE_ON_EXIT=FALSE;CIPHER=AES`
- applikationen ska vägra starta om den är konfigurerad att exponera H2 över nätverk eller lokal TCP
- verifikationer i status `BOOKED` får inte kunna ändras eller tas bort via normalt applikationsflöde
- rättelser ska ske genom korrigeringsverifikation, aldrig genom överskrivning av originaldata
- `audit_log` ska innehålla tillräckligt underlag för att visa vem eller vilken process som gjort en bokföringskritisk operation
- systemet ska spara checksummor för bilagor, rapportarkiv, backup-manifest och importjobb

### Integritetskontroller

- bokförda verifikationer ska kunna integritetsskyddas med hashkedja eller motsvarande kontrollsummebaserad kedjning
- `audit_log` ska kunna integritetsskyddas med egen hashkedja eller dagliga sammanfattningshashar
- applikationen ska verifiera integriteten vid uppstart
- applikationen ska verifiera integriteten före SIE-export, PDF-arkivering och årsstängning
- avvikelse i integritetskontroll ska ge blockerande varning eller startspärr beroende på allvarlighetsgrad

### Operativa kontroller

- backup ska innehålla manifest och checksummor så att restore kan användas för att återskapa känd god version
- systemdokumentationen ska uttryckligen ange att direkt ändring av H2-filen utanför applikationen är otillåten
- driftmiljön ska rekommenderas använda dedikerad OS-användare eller motsvarande katalogisolering
- långtidsbevarande ska stödjas genom format och metadata som kan återskapas och verifieras i minst sju år
- om högre skyddskrav krävs än vad lokal H2 med filsystemskydd kan ge, ska arkitekturen flyttas till serverstyrd databasåtkomst

## Legala och revisionsmässiga krav

Följande måste uttryckligen finnas i systemet för att det ska vara ett seriöst bokföringsprogram:

- **Löpnumrering**: obruten serie per verifikationsserie och räkenskapsår
- **Oföränderlighet**: bokförda verifikationer får inte raderas eller skrivas över
- **Rättelser**: ändringar efter bokföring ska ske med korrigeringsverifikation
- **Fullständig verifikation**: datum, text, belopp, konton och hänvisning till underlag
- **Behandlingshistorik**: logg över skapande, bokföring, korrigering, låsning, import/export och backup
- **Systemdokumentation**: beskrivning av nummerserier, tabeller, bokföringsflöden, import/export och backup-rutin
- **Spårbarhet**: från rapportrad till verifikation och från verifikation till bilaga
- **Bevarandetid**: underlag och rapporter ska kunna bevaras i minst sju år
- **Periodlåsning**: stängda perioder och räkenskapsår ska inte kunna ändras
- **Bokslutsfrist**: systemet ska varna om årsbokslut inte upprättats inom sex månader efter räkenskapsårets slut

Detta är en viktig komplettering jämfört med ursprungsskissen, som saknade explicit behandlingshistorik, systemdokumentation, periodlåsning och korrigeringsflöde.

## Acceptanskriterier

V1 är klar först när följande kan demonstreras:

1. En ny databas kan initieras med företagsuppgifter och ett räkenskapsår.
2. BAS-kontoplan kan importeras och ingående balanser kan registreras.
3. Verifikationer kan bokföras med obruten nummerserie och korrekt debet/kredit-balans.
4. Bokförda verifikationer kan korrigeras men inte ändras direkt.
5. Bilagor kan kopplas till verifikationer och är spårbara.
6. Grundbok, huvudbok, provbalans, resultatrapport, balansrapport och momsrapport kan genereras som PDF.
7. En momsperiod kan stängas och därefter blockera otillåtna ändringar.
8. Aktivt år kan exporteras till SIE4 och importeras tillbaka i testmiljö.
9. Årsstängning kan skapa nästa års ingående balanser.
10. Backup och återställning fungerar för både databas och bilagearkiv.
11. Release-builds kan produceras för Windows, Linux och macOS med fungerande launcher respektive plattformsintegration.

- teststruktur, JUnit 6.0.3 kompletterat med groovier-junit och golden-master-tester
- distributioner för Windows, Linux och macOS
- gemensam UI-valideringsstrategi och tydligare driftkrav
- acceptanskriterier för att avgöra när v1 faktiskt är klar
