# v1.0.0 Multi-Company Implementation Checklist

Detta dokument översätter migrationsplanen till en genomförbar arbetsordning.

Målet är att göra flerföretagsstödet implementerbart i tydliga batcher där varje steg går att verifiera innan nästa steg påbörjas.

## Arbetsprinciper

- gör schemaändringar först
- håll varje migration så liten och verifierbar som möjligt
- bryt inte releaseflödet i onödan mellan steg
- ändra interna kontoreferenser till `account_id` konsekvent, inte delvis
- uppdatera tester i samma batch som datamodellen ändras

## Batch 1: Grund för företag

### Mål

Införa `company` som riktig rotentitet och bära in `company_id` i de viktigaste tabellerna utan att ännu göra full UI-cutover.

### Leverabler

1. Ny migration `V13__multi_company_foundation.sql`.
2. Ny domänmodell `Company`.
3. Ny tjänst `CompanyService`.
4. `company_id` på centrala tabeller.

### SQL-arbete

Skapa `company` och migrera nuvarande singletonföretag från `company_settings`.

Lägg till `company_id` på:

- `fiscal_year`
- `account`
- `opening_balance`
- `voucher`
- `voucher_line`
- `attachment`
- `audit_log`
- `vat_period`
- `report_archive`
- `import_job`
- `closing_entry`

Fyll alla befintliga rader med första företagets id.

Sätt `not null` först efter att backfill är klar.

### Kodarbete

- skapa `Company.groovy`
- skapa `CompanyService.groovy`
- låt `CompanySettingsService` sluta vara singletonorienterad
- introducera läs/list/create för företag

### Verifiering

- databas migrerar från gammal singelföretagsmodell
- ett företag kan läsas från nya `company`
- gammal data finns kvar och pekar på rätt `company_id`

## Batch 2: Kedjehuvuden per företag

### Mål

Bryta de globala singletonkedjorna för verifikationer och audit-logg.

### Leverabler

1. Ny migration `V14__company_chain_heads.sql`.
2. Företagsspecifika chain heads.
3. Uppdaterade integritetskontroller.

### SQL-arbete

Bygg om:

- `voucher_chain_head`
- `audit_log_chain_head`

till tabeller med `company_id` som nyckel.

Migrera existerande kedjehuvud till första företaget.

### Kodarbete

Uppdatera:

- `VoucherService.validateIntegrity()`
- `AuditLogService.validateIntegrity()`
- kod som läser/skriver chain heads

### Verifiering

- hashkedjan för det migrerade företaget är fortsatt giltig
- integritetskontroll passerar efter migrering

## Batch 3: Kontonormalisering

### Mål

Göra `account` företagsspecifik och flytta alla interna relationer till `account_id`.

### Leverabler

1. Ny migration `V15__account_id_normalization.sql`.
2. `account.id` som PK.
3. `unique(company_id, account_number)`.
4. Uppdaterade referenser i beroende tabeller.

### SQL-arbete

Skapa ny konto-tabellstruktur:

- `account.id`
- `account.company_id`
- `account_number` unikt per företag

Byt beroende tabeller till `account_id`:

- `opening_balance`
- `voucher_line`
- `closing_entry`

För `voucher_line` rekommenderas att behålla:

- `account_number`
- `account_name`

som bokföringssnapshot, men relationen ska gå via `account_id`.

För `closing_entry` byt:

- `account_number` -> `account_id`
- `counter_account_number` -> `counter_account_id`

### Kodarbete

Uppdatera domänmodeller:

- `Account`
- `OpeningBalance`
- `VoucherLine`
- `ClosingEntry`

Uppdatera tjänster:

- `AccountService`
- `VoucherService`
- `ClosingService`
- `ReportDataService`
- `SieImportExportService`

### Viktiga regler

- uppslag på konto i UI får ske via kontonummer inom valt företag
- interna joins får inte längre göras via kontonummer
- SIE och rapporter får fortsätta visa kontonummer

### Verifiering

- två företag kan båda ha konto `1930`
- `voucher_line.account_id` pekar på rätt företags konto
- bokslutsposter använder `account_id`
- ingående balanser använder `account_id`

## Batch 4: Service-lager cutover

### Mål

Göra alla centrala tjänster företagsskopade.

### Leverabler

Alla affärskritiska tjänster tar explicit hänsyn till aktivt företag.

### Ordning

1. `FiscalYearService`
2. `AccountingPeriodService`
3. `AccountService`
4. `VoucherService`
5. `AttachmentService`
6. `AuditLogService`
7. `VatService`
8. `ReportDataService`
9. `ReportArchiveService`
10. `JournoReportService`
11. `SieImportExportService`
12. `ClosingService`
13. `BackupService`
14. `StartupVerificationService`
15. `SystemDiagnosticsService`
16. `SystemDocumentationService`

### Beslut för API-design

För `v1.0.0` ska service-lagret i första hand använda explicit `companyId` i publika metoder där det är rimligt.

UI-lagret kan ha en separat aktiv-företagskontext, men affärslogiken ska inte vila på dold trådlocal-state.

### Verifiering

- inga serviceanrop läser data globalt
- alla listningar, uppslag och mutationer är företagsskopade

## Batch 5: UI för aktivt företag

### Mål

Användaren ska kunna skapa och välja företag i samma installation.

### Leverabler

1. Företagsdialog eller företagspanel.
2. Aktivt företag i huvudfönstret.
3. Omladdning av vyer vid företagsbyte.

### UI-arbete

- lägg till företagshantering i `MainFrame`
- ersätt nuvarande företagsinställningsflöde med företagsskapande/redigering
- visa aktivt företag tydligt i titel/status/översikt
- blockera bokföringsflöden tills ett företag är valt

### Verifiering

- användaren kan skapa företag A och B
- användaren kan växla mellan dem
- tabeller och sammanfattningar laddas om korrekt

## Batch 6: Rapporter, moms, SIE och bokslut

### Mål

Slutföra de mest affärskritiska flödena efter flerföretagscutover.

### Delar

#### Rapporter

- `ReportDataService`
- `JournoReportService`
- `ReportArchiveService`

Krav:

- endast data från valt företag
- företagsspecifika lagringsvägar i rapportarkiv

#### Moms

- `VatService`
- momsperioder ska genereras per företag
- momsrapport ska vara företagsskopad

#### SIE

- `SieImportExportService`
- importjobb ska bära `company_id`
- export ska aldrig blanda företag
- dubblettskydd ska vara per företag

#### Bokslut

- `ClosingService`
- bokslut och nästa års ingående balanser ska vara företagsspecifika

### Verifiering

- rapport i företag A innehåller inte företag B
- momsöverföring i företag A påverkar inte företag B
- SIE-export i företag B innehåller inte företag A
- bokslut i företag A skapar inte data i företag B

## Batch 7: Testcutover

### Mål

Bygga om testsviten så att flerföretagsstödet verifieras systematiskt.

### Enhetstester

Lägg till eller uppdatera tester för:

- konto-uppslag via `account_id`
- konto-uppslag per företag
- nummerserier per företag
- chain heads per företag

### Integrationstester

Lägg till scenarier där två företag samexisterar:

- samma BAS-kontonummer i båda
- verifikationer i båda
- momsperioder i båda
- rapporter i båda
- SIE-export i båda

### Acceptanstest

Skapa ett tvåföretagsflöde som minst verifierar:

1. skapa företag A
2. skapa företag B
3. importera kontoplan i båda
4. bokför transaktion i båda
5. generera rapport i A
6. exportera SIE i B
7. kör bokslut i A
8. verifiera att ingen data läcker mellan företagen

## Batch 8: Releasepaketering

### Mål

Få releaseverifieringen grön igen efter de stora modelländringarna.

### Arbete

- fixa `jpackage`-argumenten
- fixa konfigurationscacheproblemet eller stäng av cache för release tasks
- verifiera `:app:verifyCurrentPlatformRelease`
- säkerställ att `release.sh` och README stämmer med verkligt flöde

### Verifiering

- `./gradlew build`
- `./gradlew :app:verifyCurrentPlatformRelease`

## Stop points

Efter varje batch ska följande göras:

- kör relevanta tester
- verifiera migration på en gammal singelföretagsdatabas
- granska schema och indexdiffar
- kontrollera att inga globala singletonantaganden finns kvar i ny kod

## Definition of Done

Checklistan är klar när:

- flerföretagsmigrering fungerar från befintlig databas
- `account_id` används som intern kontorelation överallt
- UI kan växla aktivt företag
- huvudboksflöden är isolerade per företag
- testerna täcker två företag i samma databas
- releaseverifieringen passerar
