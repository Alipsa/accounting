# v1.0.0 Release-Ready Plan

> **Status: alla releaseblockerare är genomförda.**

Detta dokument samlar de återstående arbeten som måste vara klara för att `v1.0.0` ska kunna betraktas som release-ready för ett svenskt bokföringsprogram för små företag.

Målet är inte att göra systemet till ett komplett affärssystem. Fokus ligger på:

- fler företag i samma installation
- löpande bokföring
- moms
- rapporter
- SIE
- årsstängning
- fungerande releasepaketering

Följande ligger uttryckligen utanför denna releaseplan:

- fakturering
- lön
- bankintegration
- årsredovisningsflöden
- anläggningsregister
- förbättrad kraschsäker bilagehantering

De två sista punkterna är uppskjutna till `v1.1.0` enligt [v1.1.0-roadmap](/home/per/project/groovy-projects/accounting/req/v1.1.0-roadmap.md).

## Styrande beslut

### 1. Flerföretagsstöd ska lösas i databasschemat

`v1.0.0` ska stödja flera företag i samma installation och samma databas.

Lösningen ska vara en riktig domänmodell för fler företag, inte en workaround via separata hemkataloger eller separata databasfiler per företag.

### 2. Lösningen ska baseras på explicit `company_id`

Vi väljer **en delad databas med explicit `company_id`** i modellen, inte separata SQL-scheman per företag.

Skäl:

- tydligare domänmodell
- mindre dold runtime-state än `set schema`
- bättre portabilitet än H2-specifika schemalösningar
- enklare att förstå, testa och underhålla långsiktigt

### 3. Konton ska göras företagsspecifika på riktigt

Nuvarande `account`-tabell har global primärnyckel på `account_number`. Det räcker inte för flerföretagsstöd i en delad databas.

För att få en hållbar modell ska konton göras företagsspecifika genom att:

- införa ett surrogate key-fält `account.id`
- lägga `company_id` på `account`
- göra `account_number` unikt per företag i stället för globalt

Detta är mer hållbart än att försöka bära hela modellen med sammansatta nycklar på kontonummer.

För `v1.0.0` är full normalisering till `account_id` ett krav, inte ett framtida förbättringsspår. Eftersom ingen release ännu har gjorts finns ingen bakåtkompatibilitet att skydda, och det är därför bättre att göra denna fundamentala datamodellsändring nu än senare.

## Releaseblockerare

Följande punkter ska vara klara innan release.

### A. Flerföretagsstöd i datamodellen (genomförd)

#### Mål

Det ska gå att skapa, välja och arbeta med flera företag i samma installation utan risk för sammanblandning av data.

#### Nödvändiga schemaändringar

1. Ersätt eller komplettera `company_settings` med en riktig `company`-tabell.
2. Lägg `company_id` på alla företagsspecifika root-tabeller.
3. Flytta bort antagandet att `company_settings.id = 1`.
4. Gör kedjehuvuden för verifikationer och audit-logg företagsspecifika.
5. Gör lagringsvägar för rapporter, backupunderlag och filarkiv företagsspecifika.

#### Tabeller som ska ses över

- `company_settings`
- `fiscal_year`
- `accounting_period`
- `account`
- `opening_balance`
- `voucher_series`
- `voucher_chain_head`
- `voucher`
- `voucher_line`
- `attachment`
- `audit_log`
- `audit_log_chain_head`
- `vat_period`
- `report_archive`
- `import_job`
- `closing_entry`

#### Rekommenderad modell

- `company(id, company_name, organization_number, default_currency, locale_tag, vat_periodicity, created_at, updated_at, active)`
- `fiscal_year.company_id`
- `account.company_id`
- `voucher_chain_head.company_id`
- `audit_log_chain_head.company_id`

Relationer via `fiscal_year_id` räcker inte som enda isolering. Där det är praktiskt och säkrare ska `company_id` även finnas direkt på tabeller som ofta filtreras eller verifieras separat.

#### Särskilt viktig designfråga: kontoplan

Kontoplanen måste vara företagsspecifik. Två företag ska kunna ha samma BAS-kontonummer utan delad rad.

Plan:

1. Lägg till `account.id` som PK.
2. Lägg till `company_id` på `account`.
3. Gör `unique(company_id, account_number)`.
4. Migrera alla interna referenser från `account_number` till `account_id`.
5. Behåll `account_number` som affärsnyckel och visningsvärde, inte som intern relationsnyckel.

#### Migreringsstrategi från dagens modell

Det måste finnas en framåtmigrering från befintlig singelföretagsdatabas.

Minimikrav:

1. Skapa första `company`-raden från nuvarande `company_settings`.
2. Sätt samma `company_id` på alla befintliga rader som hör till det enda gamla företaget.
3. Bygg nya constraints efter att data fyllts på.
4. Verifiera att hashkedjor och rapportarkiv fortsätter vara giltiga efter migreringen.

#### Service-lager

Alla tjänster ska göras företagsskopade:

- `CompanySettingsService`
- `FiscalYearService`
- `AccountingPeriodService`
- `AccountService`
- `VoucherService`
- `AttachmentService`
- `AuditLogService`
- `VatService`
- `JournoReportService`
- `ReportDataService`
- `ReportArchiveService`
- `BackupService`
- `ClosingService`
- `SieImportExportService`
- `StartupVerificationService`
- `SystemDiagnosticsService`
- `SystemDocumentationService`

Varje läsning och skrivning måste ske inom vald `company_id`. Inga queries får längre förutsätta global singleton-data.

#### UI

UI:t ska få ett tydligt aktivt företag.

Minimikrav:

1. Startdialog eller väljare för företag.
2. Möjlighet att skapa nytt företag.
3. Tydlig visning av aktivt företag i huvudfönstret.
4. Alla paneler arbetar mot aktivt företag.
5. Byte av företag ska ladda om vyer och tjänstekontext utan att blanda data.

#### Definition of Done

- Två företag kan skapas i samma installation.
- De kan ha egna räkenskapsår, kontoplaner, verifikationer, momsperioder och rapportarkiv.
- Sökning, rapporter, SIE-export och bokslut för företag A visar aldrig data från företag B.
- Migrering från gammal singelföretagsdatabas fungerar.

### B. Releasepaketering måste fungera lokalt (genomförd)

#### Mål

Den dokumenterade releaseverifieringen ska fungera på riktigt.

#### Problem som måste lösas

- `:app:verifyCurrentPlatformRelease` fallerar i nuvarande Gradle/jpackage-flöde.
- `--license-file` används tillsammans med `app-image`, vilket inte fungerar för aktuell `jpackage`.
- konfigurationscachefel förekommer i releaseuppgifterna

#### Arbete

1. Rensa upp `jpackage`-argument per pakettyp.
2. Gör release tasks kompatibla med eller uttryckligen undantagna från konfigurationscache.
3. Säkerställ att `:app:verifyCurrentPlatformRelease` passerar på aktuell plattform.
4. Säkerställ att README beskriver det verkliga releaseflödet.
5. Klargör relationen mellan lokalt releaseflöde, GitHub Actions och `release.sh`.

#### Definition of Done

- `./gradlew build` passerar.
- `./gradlew :app:verifyCurrentPlatformRelease` passerar.
- README beskriver ett verifierat releaseflöde.

### C. Momsperiodicitet måste vara tydligt färdigställd (genomförd)

#### Mål

Produkten ska antingen stödja de momsperioder som småföretag realistiskt behöver, eller vara tydligt avgränsad före release.

#### Nuvarande läge

Systemet stöder idag `MONTHLY` och `ANNUAL`, men inte `QUARTERLY`.

#### Arbete

1. Bedöm om kvartalsmoms ska stödjas i `v1.0.0`.
2. Om ja: implementera `QUARTERLY` i domän, periodgenerering, UI, tester och rapportflöde.
3. Om nej: skriv uttryckligen i README och användarmanual att `v1.0.0` endast stöder månads- och årsmoms.

#### Rekommendation

Kvartalsmoms bör stödjas i `v1.0.0` eftersom det är ett vanligt upplägg för små svenska företag.

#### Beslut

v1.0.0 stöder månads-, kvartals- och årsmoms.

#### Definition of Done

Antingen:

- kvartalsmoms fungerar fullt ut

eller:

- begränsningen är tydligt dokumenterad och accepterad som release scope

## Sekundära men viktiga uppgifter

### Dokumentation av produktens scope

README och användarmanual ska förtydliga att produkten är:

- ett bokföringsprogram
- för svenska småföretag
- med stöd för flera företag i samma installation
- inte ett komplett affärssystem

Det ska också förklaras att verifikationers underlag kan hanteras utanför programmet, så länge spårbarheten via verifikationsnummer och ordnad arkivering upprätthålls.

### Systemdokumentation för flerföretagsmodellen

Systemdokumentationen ska uppdateras så att den beskriver:

- aktivt företag
- hur företag isoleras i databasen
- företagsspecifika nummerserier
- företagsspecifika hashkedjor
- företagsspecifika arkiv och exporter

## Rekommenderad genomförandeordning

1. Besluta slutlig flerföretagsmodell och skriv migrationsdesign.
2. Implementera schemaändringar och migrering.
3. Gör service-lagret företagsskopat.
4. Lägg till UI för att skapa och välja företag.
5. Kör och utöka enhets-, integrations- och acceptanstester för två företag i samma databas.
6. Implementera eller avgränsa kvartalsmoms.
7. Fixa releasepaketeringen.
8. Uppdatera README, manual och systemdokumentation.

## Testkrav före release

Minst följande ska verifieras:

- två företag i samma databas kan använda samma kontonummer utan konflikt
- interna relationer använder `account_id` och inte globalt kontonummer
- verifikationsserier börjar om per företag och räkenskapsår
- SIE-export för ett företag innehåller bara det företagets data
- momsrapport för ett företag påverkas inte av transaktioner i ett annat företag
- bokslut för ett företag skapar inte ingående balanser i något annat företag
- backup/restore verifieras för flerföretagsdatabasen
- startup-verifiering och integritetskontroller fungerar per företag
- release smoke test passerar

## Releasebeslut

`v1.0.0` är release-ready först när:

- flerföretagsstödet är infört i den riktiga datamodellen
- releasepaketeringen fungerar
- momsperiodiciteten är färdigställd eller uttryckligen avgränsad
- dokumentationen stämmer med verklig funktion
