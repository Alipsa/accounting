# v1.0.0 Multi-Company Migration Plan

Detta dokument bryter ned flerföretagsstödet i en konkret teknisk migrationsplan för `v1.0.0`.

Planen utgår från följande beslut:

- flera företag ska stödjas i **samma installation**
- flera företag ska stödjas i **samma databas**
- lösningen ska bygga på explicit `company_id`
- lösningen ska vara hållbar och begriplig, inte bero på H2-specifika schematrick

## Målbild

Efter migreringen ska systemet stödja:

- flera företag i samma databas
- separata räkenskapsår per företag
- separata kontoplaner per företag
- separata nummerserier per företag
- separata verifikationskedjor per företag
- separata auditkedjor per företag
- rapporter, moms, SIE och bokslut utan dataläckage mellan företag

## Övergripande design

### Ny rotentitet: `company`

Inför en riktig `company`-tabell som blir ägare till all bokföringsdata.

Föreslagen struktur:

```sql
create table company (
    id bigint generated by default as identity primary key,
    company_name varchar(200) not null,
    organization_number varchar(64) not null,
    default_currency varchar(3) not null,
    locale_tag varchar(32) not null,
    vat_periodicity varchar(20) not null,
    active boolean not null default true,
    created_at timestamp not null,
    updated_at timestamp not null,
    constraint uq_company_orgnr unique (organization_number)
);
```

`company_settings` ska avvecklas efter migrering, eller reduceras till installationsnivå om det behövs för globala appinställningar. Företagsspecifika fält ska inte längre ligga där.

### Aktivt företag i applikationen

Applikationen ska arbeta med ett explicit aktivt företag i runtime.

Rekommenderad modell:

- en liten `CompanyContext` som innehåller `companyId`
- UI väljer aktivt företag
- service-anrop får `companyId` explicit eller via en smal, välkontrollerad kontext

För `v1.0.0` bör vi undvika dold trådlocal-state så långt möjligt. Explicit parameter är säkrast, men en central `CompanyContextService` kan användas i UI-lagret för att hålla aktivt företag.

## Migrationsstrategi

### Grundprincip

Migreringen ska vara **framåtriktad** från dagens singelföretagsdatabas.

Det finns redan ett implicit företag i databasen. Migreringen ska:

1. skapa en första rad i `company`
2. koppla all befintlig data till det företaget
3. bygga nya constraints och index för flerföretagsmodellen
4. lämna systemet i ett tillstånd där nya företag kan skapas utan specialfall

### Rekommenderad implementation i faser

#### Fas 1: Introducera `company`

Lägg till ny migration, exempelvis `V13__multi_company_foundation.sql`, som:

1. skapar tabellen `company`
2. migrerar data från `company_settings` till en första `company`-rad
3. lägger till `company_id` som nullable på utvalda tabeller
4. fyller `company_id` på all befintlig data
5. gör kolumnerna `not null`
6. skapar nya index

#### Fas 2: Gör tjänstelagret företagsskopat

När `company_id` finns i databasen ska service-lagret ändras så att all åtkomst filtrerar på företag.

#### Fas 3: Lägg till UI-stöd för företag

När service-lagret är företagsskopat kan UI få stöd för:

- skapa företag
- lista företag
- välja aktivt företag
- byta företag utan omstart

#### Fas 4: Städning

När allt fungerar:

- avveckla kvarvarande singleton-antaganden
- ta bort gammal logik kring `company_settings.id = 1`
- uppdatera dokumentation och tester

## Tabell-för-tabell-plan

### 1. `company_settings`

#### Nuvarande läge

`company_settings` är singleton med `check (id = 1)`.

#### Mål

Företagsspecifika inställningar ska ligga i `company`.

#### Åtgärd

- skapa `company`
- migrera innehållet från `company_settings`
- behåll `company_settings` endast om vi behöver installationsglobala inställningar senare
- annars låt tabellen vara kvar temporärt under `v1.0.0`, men sluta använda den i koden

#### Kodpåverkan

- `CompanySettingsService` måste bli en riktig `CompanyService` eller delas upp i:
  - `CompanyService`
  - eventuellt separat `ApplicationSettingsService`

### 2. `fiscal_year`

#### Mål

Varje räkenskapsår ska tillhöra ett företag.

#### Åtgärd

- lägg till `company_id bigint`
- fyll `company_id` från den migrerade standardfirman
- lägg FK till `company(id)`
- lägg index på `(company_id, start_date, end_date)`
- ändra överlappsregler så att överlapp bara förbjuds **inom samma företag**

#### Nya constraints

- unikhet behövs inte nödvändigtvis på namn, men all validering måste vara företagsskopad

### 3. `accounting_period`

#### Mål

Ingen direkt affärsrelation till företag krävs om `fiscal_year_id` alltid används korrekt, men direkt `company_id` kan ge säkrare queries.

#### Rekommendation

För `v1.0.0` räcker det att låta `accounting_period` ärva företag via `fiscal_year_id`, men läsningar ska i praktiken joina mot `fiscal_year` när företag behöver verifieras.

#### Alternativ

Om vi vill förenkla många queries kan vi även lägga `company_id` direkt här. Det är inte nödvändigt för första cuten.

### 4. `account`

#### Detta är den viktigaste strukturella ändringen.

#### Nuvarande läge

`account_number` är global primärnyckel.

#### Problem

Det fungerar inte hållbart när två företag ska ha samma kontoplan.

#### Mål

Konton ska vara företagsspecifika.

#### Rekommenderad design

Bygg om till:

```sql
create table account (
    id bigint generated by default as identity primary key,
    company_id bigint not null,
    account_number varchar(10) not null,
    account_name varchar(255) not null,
    account_class varchar(32),
    normal_balance_side varchar(16),
    vat_code varchar(32),
    active boolean not null default true,
    manual_review_required boolean not null default false,
    classification_note varchar(255),
    created_at timestamp not null,
    updated_at timestamp not null,
    constraint fk_account_company foreign key (company_id) references company(id) on delete cascade,
    constraint uq_account_company_number unique (company_id, account_number)
);
```

#### Migrering

Eftersom H2 kan vara begränsat för komplex PK-ombyggnad bör vi använda en säker tvåstegsmigrering:

1. skapa `account_new`
2. kopiera data från `account` med samma `company_id`
3. uppdatera beroende tabeller
4. byt namn från gammal till ny tabell

#### Beroenden som måste ses över

- `opening_balance`
- `voucher_line`
- `closing_entry`
- alla queries i service-lagret som slår upp konto på `account_number`

#### Viktigt beslut

För `v1.0.0` ska alla interna referenser normaliseras till `account_id`.

Det innebär att `account_number` fortsätter vara affärsnyckel, import-/exportvärde och det som användaren ser, men inte längre den interna relationsnyckeln i databasen.

Detta beslut tas nu eftersom:

- ingen publik release ännu finns
- ingen bakåtkompatibilitet behöver bevaras
- flerföretagsstödet annars riskerar att byggas på en för svag grund

### 5. `opening_balance`

#### Mål

Ingående balans får bara peka på konto inom samma företag.

#### Åtgärd

- ersätt `account_number` med `account_id`
- lägg till `company_id`
- gör PK till exempelvis `(fiscal_year_id, account_id)` eller separat surrogate key plus unik constraint
- lägg FK till `account(id)`
- verifiera att konto och räkenskapsår hör till samma företag

#### Rekommendation

Lägg `company_id` även här för att förenkla validering, indexering och skydd mot feljoiner.

### 6. `voucher_series`

#### Mål

Verifikationsserier ska vara separata per företag och räkenskapsår.

#### Åtgärd

- indirekt företag via `fiscal_year_id` kan räcka
- behåll unikhet per `(fiscal_year_id, series_code)`

#### Extra kontroll

Service-lagret måste verifiera att valt räkenskapsår tillhör aktivt företag.

### 7. `voucher_chain_head`

#### Nuvarande läge

Singleton-tabell för hela databasen.

#### Problem

Hashkedjan blir fel om flera företag delar samma huvud.

#### Mål

En kedjehuvudrad per företag.

#### Ny design

```sql
create table voucher_chain_head (
    company_id bigint primary key,
    last_content_hash varchar(64),
    updated_at timestamp not null default current_timestamp,
    constraint fk_voucher_chain_head_company
        foreign key (company_id) references company(id) on delete cascade
);
```

#### Migrering

- skapa en rad för det befintliga företaget
- sätt befintligt `last_content_hash` på den raden

### 8. `voucher`

#### Mål

Verifikationer ska isoleras per företag.

#### Åtgärd

`voucher` kan i teorin ärva företag via `fiscal_year_id`, men för säkerhet och enklare queries rekommenderas att lägga `company_id` direkt även här.

Nytt:

- `company_id bigint not null`
- index `(company_id, accounting_date)`
- index `(company_id, status)`
- unikhet för `voucher_number` ska vara företagsskopad

#### Rekommenderade unika constraints

- `unique(company_id, voucher_number)`
- `unique(fiscal_year_id, voucher_series_id, running_number)` kan behållas

#### Motiv

Direkt `company_id` gör:

- integritetskontroller enklare
- rapportering snabbare
- skydd mot framtida querymissar bättre

### 9. `voucher_line`

#### Mål

Verifikationsrader får bara använda konton inom samma företag som verifikationen.

#### Åtgärd

- ersätt `account_number` som relationsnyckel med `account_id`
- behåll gärna `account_number` och `account_name` som denormaliserad bokföringssnapshot om det behövs för revisionsspår och historisk rapportering
- lägg till `company_id`
- indexera `(company_id, account_id)`
- verifiera i service att radens konto hör till verifikationens företag

#### Rekommenderad struktur

Minst:

- `voucher_id`
- `account_id`
- bokföringsradens belopp

Valfritt men rekommenderat för historisk robusthet:

- `account_number`
- `account_name`

Det gör att gamla verifikationer fortsatt kan återges korrekt även om kontonamn senare ändras.

### 10. `attachment`

#### Mål

Bilagor ska fortsätta följa verifikationer utan korskoppling mellan företag.

#### Åtgärd

- `voucher_id` räcker funktionellt
- lägg ändå gärna `company_id` direkt för enklare sökning och lagringsstruktur

#### Lagringsvägar

Byt lagringsmönster från:

- `voucher-<id>/...`

till exempelvis:

- `company-<companyId>/voucher-<id>/...`

Detta gäller även rapportarkiv och eventuellt backuprelaterade filstrukturer där företagsspecifik segmentering behövs.

### 11. `audit_log`

#### Mål

Auditloggen ska vara företagsspecifik.

#### Åtgärd

- lägg till `company_id`
- fyll från associerad verifikation/fiscal year/attachment
- gör `company_id not null`
- index `(company_id, created_at)`
- index `(company_id, event_type, created_at)`

#### Motiv

Detta gör:

- visning av logg per företag enkel
- kedjevalidering per företag möjlig
- systemdokumentation renare

### 12. `audit_log_chain_head`

#### Nuvarande läge

Singleton för hela databasen.

#### Mål

En auditkedja per företag.

#### Ny design

```sql
create table audit_log_chain_head (
    company_id bigint primary key,
    last_entry_hash varchar(64),
    updated_at timestamp not null default current_timestamp,
    constraint fk_audit_log_chain_head_company
        foreign key (company_id) references company(id) on delete cascade
);
```

### 13. `vat_period`

#### Mål

Momsperioder ska vara isolerade per företag.

#### Åtgärd

Indirekt företag via `fiscal_year_id` räcker i teorin, men direkt `company_id` rekommenderas om vi ändå rör tabellen.

Minimikrav:

- all periodgenerering använder valt företags räkenskapsår
- momsrapport och momsöverföring filtrerar strikt via företag

### 14. `report_archive`

#### Mål

Rapportarkiv ska kunna filtreras och verifieras per företag.

#### Åtgärd

- lägg till `company_id`
- uppdatera `storage_path` till företagsspecifik struktur
- index `(company_id, created_at)`
- index `(company_id, report_type, created_at)`

### 15. `import_job`

#### Mål

SIE-importlogg ska tillhöra rätt företag.

#### Åtgärd

- lägg till `company_id`
- sätt den på importjobb från början
- checksummebaserat dubblettskydd ska sannolikt vara företagsskopat

#### Beslutspunkt

Avgör om samma fil ska få importeras till två olika företag:

- om ja: dubblettskydd ska vara per företag
- om nej: dubblettskydd kan vara globalt

Rekommendation: dubblettskydd per företag.

### 16. `closing_entry`

#### Mål

Bokslutsposter ska vara företagsspecifika.

#### Åtgärd

- lägg till `company_id`
- ersätt `account_number` med `account_id`
- ersätt `counter_account_number` med `counter_account_id`
- fyll från `fiscal_year`
- index `(company_id, created_at)`
- kontrollera att både `fiscal_year_id` och `next_fiscal_year_id` tillhör samma företag

## Indexplan

Alla nuvarande index som idag är globala behöver gås igenom.

Huvudregel:

- där läsning sker per företag ska `company_id` vara första indexkomponent

Exempel:

- `fiscal_year(company_id, start_date, end_date)`
- `account(company_id, account_number)`
- `voucher(company_id, accounting_date)`
- `voucher(company_id, status, booked_at)`
- `voucher_line(company_id, account_id)`
- `attachment(company_id, created_at)`
- `audit_log(company_id, created_at)`
- `report_archive(company_id, created_at)`
- `import_job(company_id, started_at)`
- `closing_entry(company_id, created_at)`

## Föreslagen migrationsordning i SQL

### V13: foundation

Skapa `company` och lägg till `company_id` på:

- `fiscal_year`
- `account`
- `opening_balance`
- `voucher`
- `voucher_line`
- `attachment`
- `audit_log`
- `vat_period`
- `report_archive`
- `import_job`
- `closing_entry`

Fyll samtliga rader med första företagets id.

### V14: chain heads

Bygg om:

- `voucher_chain_head`
- `audit_log_chain_head`

till företagsspecifika tabeller och initiera dem från existerande data.

### V15: account remodel

Bygg om `account` till företagsspecifik struktur med `account.id` som PK och uppdatera beroenden till `account_id`.

Detta omfattar minst:

- `opening_balance`
- `voucher_line`
- `closing_entry`

Vid behov kan arbetet delas i två fysiska migrationer, men båda ska ingå i `v1.0.0`.

### V16: constraints and indexes

Skärp:

- `not null`
- unika constraints
- foreign keys
- företagsspecifika index

### V17: cleanup

- sluta använda `company_settings` för företagsdata
- eventuellt ta bort gamla singletonconstraints
- uppdatera rapport- och filvägar

## Kodändringsplan

### 1. Domänmodeller

Inför eller ändra:

- `Company`
- `CompanySettings` eller ersätt med `Company`
- lägg `companyId` i relevanta domänmodeller

Minimikandidater:

- `FiscalYear`
- `Account`
- `Voucher`
- `VoucherLine`
- `AttachmentMetadata`
- `AuditLogEntry`
- `VatPeriod`
- `ReportArchive`
- `ImportJob`
- `ClosingEntry`

Kontorelaterade modeller ska uppdateras så att interna relationer använder `accountId`. `accountNumber` ska fortsätta finnas där den behövs som visnings- eller exportfält.

### 2. Services

Tjänsterna ska ändras i följande ordning:

1. `CompanySettingsService` till företagstjänst
2. `FiscalYearService`
3. `AccountService`
4. `VoucherService`
5. `AuditLogService`
6. `AttachmentService`
7. `VatService`
8. `ReportDataService`
9. `JournoReportService`
10. `ReportArchiveService`
11. `SieImportExportService`
12. `ClosingService`
13. `BackupService`
14. `StartupVerificationService`
15. `SystemDocumentationService`

Varje service ska få tydlig API-design för aktivt företag. Undvik halvhjärtad blandning där vissa calls är företagsskopade och andra inte.

### 3. UI

Lägg till:

- företagspanel eller företagsdialog
- väljare för aktivt företag
- uppdatering av titlar/statusrad med aktivt företag
- skydd mot att skapa data utan valt företag

## Testplan

### Enhetstester

Lägg till tester för:

- konto-uppslag per företag
- relationer via `account_id`
- nummerserie per företag
- auditkedja per företag
- verifikationskedja per företag

### Integrationstester

Lägg till tester där två företag samexisterar i samma testdatabas:

- båda har konto 1930
- båda har eget räkenskapsår
- båda bokför verifikationer i serie A
- båda skapar momsperioder
- båda exporterar SIE
- `voucher_line`, `opening_balance` och `closing_entry` refererar konto via `account_id`

### Acceptanstest

Utöka acceptanstestet till ett tvåföretagsscenario:

- skapa företag A och B
- importera kontoplan i båda
- bokför i båda
- generera rapport i A och säkerställ att B saknas
- exportera SIE i B och säkerställ att A saknas
- kör bokslut i A utan påverkan på B

## Risker

### Hög risk

- missad query utan `company_id`-filter
- ofullständig migrering från `account_number` till `account_id`
- hashkedjor som fortsätter behandlas globalt

### Medelrisk

- UI-byte av aktivt företag utan full refresh
- dubblettskydd i SIE-import som blir fel scoped
- rapportarkiv eller bilagor som återanvänder gamla lagringsvägar

### Låg risk

- dokumentationsfel
- mindre indexjusteringar

## Rekommenderade 1.0-avgränsningar

För att hålla detta genomförbart inom `v1.0.0` rekommenderas:

- håll flerföretagsstödet till en vald aktiv firma åt gången i UI
- försök inte bygga cross-company rapporter
- behåll endast det som krävs för korrekt isolering
- gör full `account_id`-normalisering nu i stället för att skjuta upp den

## Definition of Done

Flerföretagsmigreringen är klar när:

- singelföretagsdatabas kan migreras utan manuell handpåläggning
- två företag kan skapas och användas i samma databas
- samma BAS-kontonummer kan användas i båda företagen
- alla bokföringsflöden fungerar företagsskopat
- kedjor, rapporter, moms, SIE och bokslut är isolerade per företag
- testerna visar att ingen dataläckage sker mellan företag
