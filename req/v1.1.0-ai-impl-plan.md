### Arkitektur och designbeslut

#### Transportlager: intern JSON-RPC 2.0-adapter

Inget moget MCP-bibliotek för JVM/Groovy passar stdio-kravet utan att dra in ett tungt ramverk. I stället implementeras ett minimalt JSON-RPC 2.0-lager direkt med `groovy.json.JsonSlurper` och `JsonOutput`, som redan är ett beroende (`groovy-json:5.0.5`). Varje request och response är ett NDJSON-objekt (en rad per meddelande) som läses från `System.in` och skrivs till `System.out`.

#### Tre klasser med tydliga ansvar

| Klass | Ansvar |
|---|---|
| `McpServer` | Hanterar stdin/stdout-loopen och läser/skriver NDJSON. |
| `McpDispatcher` | Routar JSON-RPC-metoder, hanterar handskakningsprotokoll och översätter undantag till felkoder. |
| `AccountingMcpTools` | Implementerar alla verktyg och delegerar till befintligt servicelager. Returnerar `Map<String, Object>`. |

`McpDispatcher` hanterar `initialize`, `tools/list` och `tools/call` direkt. För `tools/call` bäddar den in verktygsresultatet i MCP:s standardformat med både `content[{type:"text", text:<json>}]` och `structuredContent: <map>` — vilket ger klienten tillgång till maskintolkbara data utan att behöva avkoda JSON-texten. Verktyg kan komplettera sin definition med `outputSchema` för schemastöd.

`AccountingMcpTools` kopplar samman verktygen med `CompanyService`, `FiscalYearService`, `AccountService`, `VoucherService`, `VatService`, `ClosingService` och `ReportDataService`. Varje privat verktygmetod returnerar en `Map<String, Object>`-karta som `McpDispatcher` sedan serialiserar.

#### Körläge via `--mode=mcp`

`AlipsaAccounting` utökas med ett `RunMode`-enum (`GUI`, `VERIFY_LAUNCH`, `MCP`) och ett nytt kommandoradsargument `--mode=mcp`. MCP-läget kör samma bootstrap som idag (loggning, databas, I18n, startup-verifiering) men hoppar över `ThemeApplier` och `MainFrame` och startar i stället `McpServer`. Vid normal avslutning (stdin stängs) anropas `DatabaseService.instance.shutdown()` och `LoggingConfigurer.shutdown()` för att frigöra H2- och loggfilshandtag.

`stdout` måste vara rent i MCP-läget: inga banners, inga Spring-loggar, inga varningsmeddelanden på standardutdata. `LoggingConfigurer.configure()` omdirigerar redan loggning till fil. I MCP-grenen i `main()` får ingenting skriva till `System.out` innan `McpServer.run()` tar över. Acceptanstestet för headless-läget skall starta applikationen som en riktig underprocess (via `ProcessBuilder`), skicka `initialize` på stdin, och verifiera att exakt en giltig JSON-RPC-rad returneras på stdout — utan extra rader, utan banner och utan loggtext.

#### MCP-protokollstöd

| Egenskap | Beslut |
|---|---|
| Protokollversion | `2025-11-25` (senaste publicerade MCP-revision vid planens uppdatering) |
| Versionsförhandling | Servern svarar alltid med `protocolVersion: "2025-11-25"` oavsett vad klienten begär. Klienten avgör om den kan fortsätta. |
| Notifikationer | `notifications/initialized` och alla övriga `notifications/*`-meddelanden ignoreras tyst (inget svar). |
| Okända metoder | Returnerar JSON-RPC -32601 Method not found. |
| Sidnumrering | Ej stödd i v1. Rapportverktyg med potentiellt stor output erbjuder en `limit`-parameter i stället. |
| Avbrutna anrop | `$/cancelRequest` ignoreras tyst. |
| Progress-notiser | Inte implementerade. |

Alla `inputSchema` och eventuella `outputSchema` i `tools/list` skall vara giltig JSON Schema 2020-12, vilket är MCP:s standarddialekt när inget explicit `$schema` anges.

#### Servertvingad förhandsgranskningstoken

Skrivoperationerna är **inte** skyddade enbart av konventionen att modellen skall ringa preview-verktyget först. Servern tvingar fram ordningen med en enkel statslös token.

**Mekanism:** `preview_voucher`, `preview_year_end` och `get_vat_report` beräknar en SHA-256-hash av en kanonisk JSON-representation av den nyttolast som är relevant för att identifiera att exakt det som förhandsgranskas är det som utförs. Hashen returneras som fältet `preview_token` i svaret.

- `post_voucher` kräver ett `preview_token`-argument. Verktyget reberäknar hashen från de inkommande argumenten och avvisar anropet om tokenen inte stämmer. Felmeddelandet skall vara `"preview_token krävs — kör preview_voucher med exakt samma argument först."`.
- `close_fiscal_year` kräver ett `preview_token`-argument beräknat av `preview_year_end`.
- `book_vat_transfer` kräver ett `report_hash`-argument beräknat av `get_vat_report` (VAT-perioden har redan ett `reportHash`-fält som kan återanvändas).
- `create_correction_voucher` kräver ingen token — det enda argumentet är `original_voucher_id` som redan identifierar exakt vilken verifikation som korrigeras.

Tokenen har ingen tidsbegränsning i v1 (ingen serverstatuslagring). Skyddet gäller mot nyttolastbyten, inte tidsgränser.

**Kanaliseringstabell:**

| Förhandsgranskning | Skrivverktyg | Token |
|---|---|---|
| `preview_voucher(args)` → `preview_token` | `post_voucher(args + preview_token)` | SHA-256 av `{fiscal_year_id, series_code, accounting_date, description, lines}` |
| `preview_year_end(fiscal_year_id)` → `preview_token` | `close_fiscal_year(fiscal_year_id, preview_token)` | SHA-256 av `{fiscal_year_id, closing_account}` |
| `get_vat_report(vat_period_id)` → `report_hash` | `book_vat_transfer(vat_period_id, report_hash)` | Beräknas av `VatService.calculateReport` |

#### `company_id` som explicit parameter

Alla verktyg som behöver företagskontext tar `company_id` som explicit argument. Ingen sessionstate finns på servernivå.


| Fil | Förändring |
|---|---|
| `AlipsaAccounting.groovy` | Nytt `RunMode`-enum, `--mode=mcp`-argument, MCP-startblock i `main()`. |
| `app/build.gradle` | `skill/`-katalogen inkluderas i alla distributionsartefakter (se nedan). |

**Distributionspaket — alla plattformar:**

| Artefakt | Metod |

| `distZip` / `distTar` | `distributions.main.contents { from(rootProject.file('skill')) { into 'skill' } }` |

### Testansats

- Integrationstester använder `DatabaseService.newForTesting()` med `@TempDir`, samma mönster som `FiscalYearServiceTest`.
- `McpDispatcherTest` verifierar JSON-RPC-protokollagret utan databas och kontrollerar att `structuredContent` är med i `tools/call`-svar.
- `AccountingMcpToolsTest` verifierar affärsregler per verktyg:
  - Stängt räkenskapsår blockerar `post_voucher`.
  - Obalanserade rader returnerar fel från `preview_voucher` och `post_voucher`.
  - `post_voucher` utan `preview_token` avvisas med tydligt felmeddelande.
  - `post_voucher` med token beräknad från annan nyttolast avvisas.
  - `book_vat_transfer` utan `report_hash` avvisas.
  - `close_fiscal_year` utan `preview_token` avvisas.
  - Okänt verktyg ger JSON-RPC -32602-felobjekt.
- `McpHeadlessStartupTest` innehåller ett processtest: startar applikationen som underprocess via `ProcessBuilder` med `--mode=mcp --home=<tempdir>`, skickar `initialize` på stdin, stänger stdin, läser alla stdout-rader och verifierar att exakt en giltig JSON-RPC-rad returneras och att inga extra rader förekommer.

### Leveransordning

1. `McpServer` + `McpDispatcher` — protokollagret med `structuredContent`.
2. `--mode=mcp` i `AlipsaAccounting` — bootstrap, ren avslutning, stdout-renhetstest.
3. Läsande verktyg i grupper (company/år → konton/verifikationer → rapporter/moms).
4. `preview_voucher` med `preview_token` → `post_voucher` med tokenvalidering → `create_correction_voucher`.
5. `get_vat_report` med `report_hash` → `book_vat_transfer` med hashvalidering.
6. `preview_year_end` med `preview_token` → `close_fiscal_year` med tokenvalidering.
7. Affärsregeltest och bypass-test som speglar GUI-beteende.
8. `skill/accounting-mcp.md` och distributionspaket (alla plattformar).

Moms och bokslut kan levereras separat efter att verifikationsflödet visat att verktygskontrakt, felhantering och skillinstruktioner fungerar i praktiken.


# Phases
- Phase 1. Protocol + headless skeleton
   Implement McpServer, McpDispatcher, --mode=mcp, initialize, tools/list, tools/call, stdout cleanliness, no real business writes.
- Phase 2. Read-only tools
   Add company, fiscal years, accounts, vouchers, trial balance, ledger, VAT period/report reads. This proves service wiring and structured MCP output without mutation risk.
- Phase 3. Voucher write flow
   Add preview_voucher, token enforcement, post_voucher, correction voucher, and bypass tests.
- Phase 4. VAT + year-end writes
   Add book_vat_transfer, preview_year_end, close_fiscal_year, and tests for closed-year/period constraints.
- Phase 5. Packaging + skill docs
   Add skill/accounting-mcp.md, Gradle distribution inclusion, and final full ./gradlew build.
