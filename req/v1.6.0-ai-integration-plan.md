# v1.6.0: Remote MCP and AI-assisted voucher workflow

## Summary

Replace the current stdio-only MCP mode with a Streamable HTTP MCP server embedded in the running desktop app. It will listen only on `127.0.0.1:48652/mcp`, require a per-installation bearer token, and stop when the app closes. The app will not call AI or web APIs directly; Claude Code, Codex, Kimi and Vibe connect as MCP clients. "Remote" here refers to the MCP *transport* (Streamable HTTP replacing the stdio-launched subprocess transport), not network reachability: accounting data must never leave the loopback interface, so the server intentionally stays bound to `127.0.0.1` rather than becoming LAN- or network-accessible.

## Server and MCP interface

- Remove `--mode=mcp` and the stdio server; start/stop the HTTP server with the GUI lifecycle. When the app is not open, the endpoint simply refuses connections—no CLI process is launched or held up.
- Implement the standard single-endpoint Streamable HTTP transport (`POST /mcp`, with the required `GET` handling) and JSON-RPC dispatcher for MCP protocol `2025-11-25`. Negotiate that version at `initialize`, issue an opaque `Mcp-Session-Id`, and require it for later requests; reject unsupported versions and unknown sessions. Do not retain legacy SSE or stdio compatibility. Implement the listener with the JDK's built-in `com.sun.net.httpserver.HttpServer` rather than adding a new HTTP framework dependency (Jetty/Netty/Javalin) — the server is a small loopback-only listener, and this keeps it consistent with the project's minimal-dependency, embedded-only philosophy (cf. H2 embedded-only).
- Add an AI settings section showing server status, the fixed endpoint, copyable bearer token/configuration, and a “regenerate token” action. Generate a cryptographically strong token on first run and store it in user preferences; this is plaintext at rest, accepted because the server is loopback-only. Reject missing or invalid `Authorization: Bearer` headers, invalid `Host` values, and any supplied `Origin` not matching the loopback endpoint before protocol dispatch. Regenerating the token invalidates all sessions immediately: clients using the old token or session are rejected on their next request, without requiring an app restart.
- Extend protocol support from tools-only to MCP resources and prompts:
  - Static resources for the accounting workflow guide and a curated source catalogue containing Bokföringstips and Skatteverket URLs.
  - Prompts for voucher guidance, VAT handling, corrections, year-end, and source-use guidance.
  - No server-side browsing or outbound web requests; clients use their own web capability and are guided to prioritize those two sources.
- Keep existing read tools and non-voucher tools (`list_vouchers`, `get_general_ledger`, `get_trial_balance`, etc.) — these already give the LLM context on prior voucher registrations and account activity when assisting with a new entry. Add `accounting_method` to `get_company_info`.
- Remove `post_voucher` from the exposed tools. Keep `preview_voucher`, then add:
  - `get_active_voucher_draft` — returns the active GUI voucher’s editable fields, line items, current company/year context, and `saved: false`.
  - `set_active_voucher_draft` — validates a complete replacement draft, activates and focuses the voucher panel, fills the GUI voucher, and returns `awaiting_user_save: true`; it never creates a database voucher. When no draft is open, it creates a new unsaved voucher draft first.
- `get_active_voucher_draft` returns a structured tool error when the voucher panel has no existing draft; it never silently creates one.
- Preserve preview-token safeguards for VAT transfer, year-end close, and SIE import. Make every such preview token a single-use, in-memory capability: atomically reserve it immediately before the guarded operation starts, including when the client later disconnects; a repeat call with that token returns a structured “already used” error and must preview again. Keep the token consumed after a successful operation, but release the reservation when its transactional service call fails so the client can safely retry. These state-changing operations run under the UI operation guard; their existing client-side confirmation guidance remains in the bundled skill.

## GUI coordination and confirmation

- Introduce a UI/MCP coordinator shared by the HTTP tools and `MainFrame`.
- Serialize state-changing MCP operations with a 16-request queue. A call is rejected with a retryable busy response when that queue is full. A request waits up to 30 seconds, including queue time; timed-out queued calls do not start, while a started operation is not interrupted and returns a 504 response explaining that it may still be completing. This avoids interrupting a database operation mid-transaction.
- Read-only calls use a maximum of eight workers and a 32-request FIFO queue. Additional requests receive a retryable busy response; read-only calls do not show the overlay.
- During a state-changing call, show a glass-pane overlay reading “AI-assistenten arbetar …” and prevent Swing interaction. Create and remove it on the EDT; remove it in `finally` when the operation completes, is rejected before starting, fails, or the app shuts down. A client disconnect does not cancel an already-started operation or remove its overlay early.
- Expose safe EDT-bound snapshot/apply methods from `VoucherPanel`; never let HTTP worker threads read or mutate Swing state directly.
- The normal GUI Save button remains the sole way to persist an AI-prepared voucher. Users can edit or discard the displayed unsaved draft before saving.

## Documentation and privacy

- Replace stdio setup instructions in README, privacy policy, bundled MCP skill, and user manual with localhost HTTP/token configuration.
- Provide copy-paste connection examples for Claude Code, Codex, Kimi and Vibe, using the shared endpoint and bearer-token header; state that the desktop app must be running.
- Document the confirmation model, nonpersistent voucher drafts, trusted-source catalogue, token regeneration, and the fact that the chosen AI client may transmit accounting data to its provider. State that an expected, harmless connection error may be logged by the AI client while the desktop app is closed.
- Add release notes and an upgrade note: stdio configuration is no longer supported, and existing AI CLI configurations must be replaced with the documented HTTP endpoint and bearer token.

## Test plan

- HTTP contract tests: authentication, Host and Origin rejection; initialization/version negotiation/session issue; required session ID and unknown-session rejection; tools/list; tools/call; resources/list/read; prompts/list/get; correct JSON-RPC and content types; quick failure when the server is stopped; and the required Streamable HTTP `GET` handling on `/mcp`.
- Client compatibility acceptance: verify initialization and one read/write workflow against the released Claude Code, Codex, Kimi, and Vibe clients using protocol `2025-11-25`. If any client cannot negotiate it, retain `2025-06-18` as a second supported version and select the client's requested version during initialization; do not ship a hard version rejection without this verification.
- Concurrency tests: state-changing calls are serialized; full mutation and read queues return a retryable busy response; a 30-second operation timeout preserves a started operation without interrupting it; up to eight reads may proceed concurrently.
- Tool tests: `get_company_info` includes accounting method; draft snapshot accurately reflects unsaved rows; draft replacement validates and updates the UI without writing a voucher; `post_voucher` is no longer discoverable.
- Swing tests: mutations execute on the EDT, applying a draft focuses the voucher panel, overlay blocks interaction only during mutations, and cleanup always removes it, including during app shutdown.
- Regression tests for all retained MCP tools, token-protected write workflows, token regeneration invalidating existing sessions, app shutdown, and full `./gradlew build`. Verify that repeating a token-guarded operation after client disconnect or successful completion is rejected without a second import, closing, or VAT transfer, while a failed transactional operation releases its reservation for retry.

## Assumptions

- The fixed loopback port is `48652`; a port collision prevents MCP startup and is surfaced in AI settings rather than silently choosing another port. The rest of the desktop app remains fully usable in this case — a port collision blocks only the MCP endpoint, not application startup.
- The server is intentionally local-only, so HTTPS, LAN discovery, account management, and cloud hosting are out of scope.
- Direct Anthropic/OpenAI/Moonshot/Mistral API integration, API-key storage, in-app chat, and server-side web search are out of scope for v1.6.0.
